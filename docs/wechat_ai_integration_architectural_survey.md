# 微信 / 企微 AI 智能体接入架构调研报告 (Architectural Survey)

**日期**: 2026-03-31
**主题**: 如何让底层 AI 大脑与微信生态无缝且智能地对接（客服 / 销售 / 分发场景）

---

## 路线对决：API 原生架构 vs OpenCLI (RPA) 外挂架构

在探讨如何接管微信沟通时，工业界有两条分歧极大的路线，其优缺点如下：

### 1. 外挂屏幕点击流 (OpenCLI / RPA / UIAutomation)
模拟人类的鼠标和键盘，通过图像识别或 DOM 操作硬控微信客户端。
*   **极大优势（零权限摩擦）**: 不需要向公司 IT 申请任何管理员权限、不要配置 Secret，甚至连个人微信号都能直接接管。
*   **致命劣势（脆弱与低效）**: 
    1. **排他性**: 脚本控制鼠标时，人类无法干预电脑。
    2. **并发灾难**: 只能“单线程”处理客户。如果 10 人同时发消息，外挂会陷入混乱，经常发生“把发给 A 的话错发给 B”的事故。
    3. **版本脆性**: 一旦微信客户端弹出一个升级框或改了 UI，整个流水线立刻崩溃。

### 2. 原生大闸架构 (企业微信官方 Webhook / 自建应用 API)
通过 HTTP `requests` 与腾讯官方的服务器直接高频互动。
*   **优势（军工级稳定）**: 毫秒级并发。能同时与 1 万个客户聊天。后台静默运行，无需电脑常亮屏幕。所有的历史记录都可以被完美地写进 SQL 数据库中。
*   **劣势（行政门槛）**: 必须是企业微信的超级管理员（或应用管理员），在 `work.weixin.qq.com` 后台获取到 `CorpID` 和 `App Secret` 才能开发。

**结论**: OpenCLI 适合前期**地下黑客式的 MVP 验证**（几天内强行跑通商业模式）；一旦进入正规实战，**必须走官方 API 架构**。

## 进阶：如何打造真正的“重型 AI Agent（智能体）”

接通大模型只是第一步（单纯发消息），要让 AI 像金牌猎头一样智能，必须拆除框架（不需要重型的 Wecom-cli 或 LangChain），采用**“原生手写 Agent (Native Agent)”**，注入以下四大核心基因：

### 1. RAG 企业外脑 (Retrieval-Augmented Generation)
*   **问题**: 大模型对公司内部独有的 SaaS 产品或猎头费率一无所知，极易发生“胡编乱造（幻觉）”。
*   **能力**: 在回答客户前，强制大模型通过 Python 脚本去搜索现有的 `notion_rag` 数据库，将内部知识嵌入提示词。

### 2. 长期记忆中枢 (Memory)
*   **问题**: 大模型是“金鱼记忆”，记不住客户上一句说了什么。
*   **能力**: 为每个企微 `userid` 在 SQLite 开辟一张独立履历表，每次对话时需强行向大模型注入前 10 轮的历史摘要。

### 3. 系统人设锁死 (System Prompting)
*   **能力**: 剥夺 AI “我是一个人工智能模型”的共性设定。将其精神钢印死死钉在“深科技投融顾问”上。禁止使用幼稚表情包，强调专业克制。

### 4. 动作执行与阻断 (Function Calling & Handoff)
*   **能力（工具）**: 客户要求面谈时，AI 能够提取时间并通过飞书 API 自动建会。
*   **能力（安全）**: 前置一个情绪识别探针，当监测到客户存在疑惑、扯皮或涉及核心高危财务数据时，AI 立刻闭麦，并给首脑（人类管理员）发送强插播警报：“请人工接管沟通”。
