# 多路分发平台过程管控架构升级计划 (Orchestrator Management Roadmap)

您的担忧非常专业且直击痛点。作为一名前线指挥官，当系统告诉你“生成了3条”，您脑海里冒出的三个灵魂拷问是：
1. **黑盒焦虑（过程透明度）**: 到底是从多少条垃圾源头里筛出来的这 3 条？有没有漏掉重要的没抓到？
2. **发布风控（审核干预）**: 辛辛苦苦做出来的小红书和公众号对品控要求极高，系统怎么敢直接无脑发出去？我怎么能先审一眼改两个错别字再发？
3. **记忆紊乱（防重发机制）**: 明天再跑一次管线，这 3 条会不会被二次捞出来又发一遍？

为了让这套粗放的“自动化脚本”真正蜕变成一套军工级的**「内容中台与过程管理系统」**，我作为产品经理与技术架构师，为您制定了以下“逐步演进”的总体规划蓝图。我们不能头痛医头脚痛医脚，而是要系统级地重构这套管线的“吞吐流程”。

## 💡 产品规划全景蓝图 (Product Roadmap)

我们要将系统打造成一条**带有安检站的汽车流水线**。

> [!IMPORTANT]
> **核心产品哲学：流水线全自动化生产 + 人类一键质检放行 (Human-in-the-loop)。**

我们将分三个阶段（Phase）逐步改造您的平台：

### 阶段一：建立“透明漏斗”数据大屏 (Pipeline Observability)
解决您的第一个担忧：“到底抓了多少，筛了多少，漏了多少？”
不再只看干瘪的命令行日志，我们要让数据开口说话。
*   **产品设计**：在总控面板的 `管线列表` 旁，新增一个**【执行追踪大盘】**。
*   **指标漏斗**：展示每次跑批的“转化率漏斗”：
    `[源头抓取数: 300条] -> [重复过滤剔除: 210条] -> [LLM相关性筛选淘汰: 87条] -> [成功生成草稿: 3条]`。
*   **价值**：如果某天漏斗第一步“抓取数”变成了 0，您立马就知道爬虫死掉了，而不是以为今天没有好新闻。

### 阶段二：建立中央“隔离核准收件箱” (Human Auditing & Approval)
解决您的发稿焦虑。我们绝对不能允许 AI 绕过您直接往小红书和微信公众号这种核心资产阵地发贴（除非您极为自信）。
*   **隔离审查机制**：在原本的流程中切上一刀，砍断“生成->直接发送”的旧流程链条。
*   **产品设计**：在前端增加一个**【草稿待审箱 (Draft Inbox)】**栏目。
*   **流程流转**：
    1. AI 管线在后台疯狂筛沙子，写好 3 篇极其惊艳的图文 Markdown 后，它把稿件**拦截**并保存在本地数据库的【待审区】。
    2. 管线只往 Telegram 里发一条提醒信息：“主人，今天为您生成了 3 篇绝佳商战热点，请移步面板审核”。
    3. 您在面板里打开草稿，可以随意修改标题或段落错别字。
    4. 确认无误后，点击【**一键齐发**】，系统再调动底层的 `wechat_adapter` 和 `xhs_adapter` 把稿子扔向全网。

### 阶段三：强化防重发引擎 (Global De-duplication)
*   **状态强一致性**：在底层给每一条抓取回来的原始新闻计算出独一无二的 MD5 身份信息（URL Hash），构建一张铁壁合围的“黑名单表”。
*   只要一条新闻进了“草稿箱”或者“已发布库”，明天的脚本哪怕再去源头抓到它 10 次，也会在第一层漏斗把它直接烧掉，彻底根绝同一份素材被反复加工的资源浪费。

## 🛠️ 技术架构落地指南 (Technical Implementation)

> [!NOTE]
> 这里的每一个步骤代表一个开发冲刺 (Sprint)。我将等待您的同意后，依次执行。

### Step 1: 改造底层数据库 (Database Schema Expansion)
目前您的 `state_manager.py` 里只有一张粗糙的 `publish_log` 分发记录表。我们需要做大手术：
1. **[NEW] `pipeline_run_history` 表**：记录每次管线执行的统揽数据（运行时间、时长、抓取总量、过滤量、生成结果）。
2. **[NEW] `content_drafts` 表**：
   * 字段包含：`draft_id`, `pipeline_id`, `title`, `markdown_body`, `status(枚举: PENDING/PUBLISHED/DISCARDED)`
   * 这张表将成为系统真正的内容存储核心。
3. **重构 `state_manager.py` 防重判定**：扩展判定维度，若抓到的 URL 在这几张表里均有记录，则直接跳出循环。

### Step 2: 截断大闸，重构管线执行代码 (Pipeline Isolation)
修改 Python 爬取与执行总线：
1. 将 `llm_engine.py` 吐出的结果直接拦住，不许向 `publishers/*_adapter.py` 转交。
2. 转而使用 `sqlite3` 将包含标题和正文的字典 `INSERT INTO content_drafts` 中。
3. 将执行结果的数据指标汇总（`total_scraped`, `total_passed_llm`）刷入 `pipeline_run_history` 表。

### Step 3: 前端控制面板开辟后院 (Next.js UI Refactoring)
全面动刀前端仪表盘 `web_dashboard/src/app`：
1. **草稿箱页面构建**：开发一个类似邮件收件箱的列表，并在右侧配上一个可供修改的 Markdown 编辑大文本框。
2. **挂载通过与否决 API**：写两个后端 FastAPI 路由 `/api/drafts/{id}/publish` 与 `/api/drafts/{id}/discard`。
3. 把被搁置已久的 `xhs_adapter` 和 `wechat_adapter` 从脚本里抽离出来，改为由这颗前端页面按钮来触发调用。
